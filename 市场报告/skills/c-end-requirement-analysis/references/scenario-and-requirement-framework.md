# 场景与需求分析框架

## 1. 场景分析四步法

1. 定义用户与目标
- 目标用户是谁，当前要完成什么任务。
- 任务成功的结果是什么，失败表现是什么。

2. 还原真实路径
- 触发 -> 进入 -> 操作 -> 反馈 -> 完成/流失。
- 记录每一步的行为成本（时间、认知、操作复杂度）。

3. 识别关键断点
- 高频断点：发生次数多。
- 高损失断点：影响转化或留存显著。
- 高情绪断点：用户抱怨集中。

4. 形成场景优先级
- 频次（Frequency）
- 痛感（Pain）
- 业务价值（Value）
- 解决复杂度（Effort）

建议用 1-5 分量化，按 `(频次 + 痛感 + 价值) / 复杂度` 排序。

## 2. 需求分析结构

每个需求项按固定结构展开：
- 问题定义：当前问题和影响范围。
- 目标定义：成功后的可衡量结果。
- 方案描述：要做什么，不做什么。
- 功能拆解：核心能力、支撑能力、边界能力。
- 信息架构影响：对象、字段、页面、导航、状态流。
- 验收标准：可测试、可判定、可回归。
- 风险与依赖：技术/流程/资源依赖，主要风险与兜底。

## 3. 信息架构分析检查单

- 对象层：实体是否清晰，命名是否统一。
- 关系层：对象之间一对一、一对多关系是否明确。
- 页面层：信息分层是否符合用户心智。
- 导航层：进入路径是否最短，返回路径是否可预期。
- 状态层：空态、加载态、错误态、异常态是否完整。
- 权限层：用户可见可操作范围是否明确。

## 4. 优先级建议

- P0：必须做，不做会阻断主链路或核心目标。
- P1：应该做，能显著提升体验或效率。
- P2：可选做，收益明确但短期不关键。
- P3：暂缓，进入观察池。

结合 RICE：
- Reach：覆盖用户规模
- Impact：单用户影响强度
- Confidence：证据可靠度
- Effort：人力投入

分数越高优先级越高，但需与版本目标一致。
