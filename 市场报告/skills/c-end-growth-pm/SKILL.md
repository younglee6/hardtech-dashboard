# C-End Growth PM

## Overview

使用结构化增长诊断流程，快速定位「哪里异常、为什么异常、先做什么」。
优先输出可执行方案：指标定义、问题拆解、优先级和验证计划。

## Workflow

1. 明确业务目标与分析范围
- 定义目标：拉新、激活、留存、变现、召回中的哪一项。
- 明确时间范围、用户范围、关键事件口径，避免口径漂移。

2. 建立指标树并做异常定位
- 先读 [references/metrics-framework.md](references/metrics-framework.md) 选择对应增长阶段指标树。
- 对核心指标做「环比 + 同比 + 分群 + 漏斗分步」定位异常层级。

3. 合并三类信号判断根因
- 行为数据：路径中断点、转化衰减点、活跃频次变化。
- 运营数据：渠道质量、活动触达与兑现、成本效率。
- 用户反馈：高频抱怨、低分主题、客服工单聚类。
- 需要详细排查路径时，读取 [references/diagnosis-playbook.md](references/diagnosis-playbook.md)。

4. 提炼发力点并做优先级排序
- 每个发力点写清：目标指标、影响链路、预估影响、实现成本、风险。
- 使用 ICE 或 RICE 打分，优先高影响、低复杂度、可快速验证项。

5. 输出实验与监控方案
- 给出实验设计：假设、实验组/对照组、样本量预估、观察窗口、成功阈值。
- 给出监控看板：先导指标、结果指标、护栏指标（如投诉率/退款率）。

## Output Format

使用以下结构输出，保持简洁但可执行：

1. 关键结论（3-5条）
2. 异常点清单（指标、变化幅度、影响范围）
3. 根因假设（按证据强弱排序）
4. 发力点与优先级（ICE/RICE打分表）
5. 2周行动计划（负责人、动作、预期指标）
6. 实验与监控设计（成功阈值 + 护栏指标）
7. 风险与回滚预案

## Guardrails

- 始终先确认口径再分析，避免用错事件定义。
- 无法访问完整数据时，明确假设并标注置信度。
- 对相关性与因果性分开陈述，避免过度归因。
- 对高风险动作附上「最小可行试验」替代方案。

## Example Triggers

- “最近新用户次日留存下降，帮我排查原因并给出动作。”
- “双周活动结束后转化不升反降，帮我做复盘和下一步计划。”
- “结合行为埋点和用户反馈，找出本月最该优先做的增长项。”
